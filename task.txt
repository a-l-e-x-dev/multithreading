## Задача 1. Java SE: многопоточный счётчик в памяти (консоль)

Цель: увидеть race condition на простом примере и исправить её разными способами.

### Требования

1. Написать консольное приложение на Java SE.
2. Есть общий объект Counter с методом increase() и get().
3. В public static void main создать 5 потоков, которые стартуют одновременно и увеличивают один и тот же счётчик (например, по 100_000 раз каждый).
4. В конце вывести:

   * ожидаемое значение (5 × количество инкрементов),
   * фактическое значение,
   * разницу (если есть).

### Что сделать студенту

* Реализовать три версии Counter:

  1. Naive: просто value++ (ожидаемо ломается под нагрузкой).
  2. Synchronized / монитор: synchronized (или synchronized(lock)).
  3. Atomic: AtomicLong (и опционально LongAdder).

### Доп. пункт (важно для понимания)

* Добавить вариант с volatile long value и объяснить, почему volatile не решает проблему инкремента.

## Задача 2. Spring Boot: счётчик через REST + параллельные клиенты

Цель: перенести проблему многопоточности на уровень веб-приложения и воспроизвести “потерянные инкременты” при параллельных запросах.

### Требования

1. Сделать Spring Boot приложение.
2. Реализовать два эндпоинта:

   * POST /increase?delta=1 — увеличивает счётчик на delta (по умолчанию 1), возвращает текущее значение.
   * GET /get — возвращает текущее значение.
3. Счётчик хранить в памяти приложения (Singleton service).

### Нагрузочная проверка

* Написать небольшой клиент/тест (можно отдельным классом в проекте или JUnit), который:

  * запускает 5 параллельных потоков,
  * каждый поток делает, например, 10_000 HTTP запросов на /increase,
  * по завершении сравнивает результат /get с ожидаемым.

### Что должен продемонстрировать студент

* Наивная реализация (без синхронизации) должна иногда давать неверный итог.
* Исправление минимум двумя способами:

  * synchronized / Lock
  * AtomicLong (опционально LongAdder)
* Коротко описать, почему именно выбранный способ корректен.

## Задача 3. Spring Boot + Embedded H2: счётчик в базе данных

Цель: показать, что “перенести значение в БД” не означает автоматически решить конкурентность — и научиться делать атомарный инкремент на уровне SQL/транзакций.

### Требования

1. Подключить embedded H2 (in-memory) и Spring Data JPA (или JDBC).
2. Создать таблицу counter (например: id, value).
3. Переделать сервис так, чтобы значение счётчика хранилось в БД, а эндпоинты остались:

   * POST /increase
   * GET /get

### Часть A — намеренно неправильный вариант

* Реализовать логику “прочитал → увеличил в Java → сохранил”:

  * findById(), value++, save()
* Запустить ту же нагрузку (5 потоков × N запросов).
* Зафиксировать, что появляются lost updates (потерянные инкременты) при конкуренции.

### Часть B — правильный вариант (атомарно)

Исправить на один из вариантов:

1. Атомарный SQL update:
   UPDATE counter SET value = value + :delta WHERE id = :id
2. Optimistic locking (@Version) + retry.
3. (Опционально) Pessimistic lock (SELECT ... FOR UPDATE).

### Приёмка

* Под нагрузкой итоговое значение всегда равно ожидаемому.
* Студент может объяснить разницу между:

  * потокобезопасностью в памяти (Java),
  * конкурентностью на уровне БД (lost update),
  * атомарным update и блокировками/версионированием.
